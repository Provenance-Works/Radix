# 設計原則

> **対象読者**: 全員

## コア哲学

Radixは、Lean 4向けの形式検証済み低レベルシステムプログラミングライブラリです。設計は明確な優先順位階層に基づいています：

```
正しさ > 安全性 > 性能 > 使いやすさ
```

あらゆる設計判断はこの階層に照らして評価されます。正しさのために性能を犠牲にすることはあっても、性能のために正しさを犠牲にすることは決してありません。

## 原則

### 1. 検証の最大化、信頼の最小化

seL4やCertiKOSに着想を得て、Radixの3層アーキテクチャは形式検証済みの部分を最大化し、信頼計算基盤（TCB）を最小化します。

- **Layer 2-3**（仕様、実装、証明）：Lean 4のカーネルにより完全に検証済み。信頼は不要。
- **Layer 1**（システムブリッジ）：Lean 4の組み込みIO APIをラップする最小限のコード。外部仕様を引用する名前付き信頼公理（`trust_*`）を含む。Radixで人間の監査が必要な唯一の部分。
- **Mathlib**：形式検証済み。TCBを拡大**しない**。

TCBは以下のみで構成：Lean 4 コンパイラ/ランタイム、`trust_*` 公理宣言、Lean 4 のデフォルト公理（`propext`、`Quot.sound`、`Classical.choice`）。

### 2. ゼロコスト抽象化（NFR-002）

型と操作は効率的な機械語にコンパイルされなければなりません：

- Radixの `UInt32` は Lean 4 組み込みの `_root_.UInt32` をラップし、これはCの `uint32_t` にコンパイルされる。ラッパーはコンパイル時に消去される。
- 全ての `@[inline]` 関数はコンパイラによりインライン化される。
- 証明はランタイムで消去される — Tier 1（証明付き）APIは Tier 2（チェック付き）APIに対してゼロオーバーヘッド。
- 符号付き型（`Int8`-`Int64`）は符号なし型をラップ — 同じストレージ、異なる解釈。符号表現のためのヒープ割り当てなし。

### 3. 純粋 Lean 4 ポリシーの厳守（C-001）

RadixはカスタムCコード、FFIバインディング、カスタムC ABIシムへの `@[extern]` 呼び出しを**厳格に禁止**しています。全てのOS対話はLean 4の組み込みIOライブラリ（`IO.FS`、`IO.Process` 等）を経由します。

**根拠:** カスタムC相互運用は、Lean 4の参照カウントランタイムとの不完全な相互作用により、メモリリーク、use-after-free、ファイルディスクリプタリサイクリングバグのリスクがある。Lean 4のマネージドIO境界内に留まることで、Radixは完全な安全性保証を維持する。

### 4. 仕様言語としてのBitVec（ADR-002）

Mathlibの `BitVec n` は、全Radix整数型の仕様レベルにおける正準表現です：

- **Layer 3**：仕様は `BitVec n` の用語で記述
- **Layer 2**：実装は性能のために Lean 4 組み込み型を使用
- **証明**：Layer 2 の操作と Layer 3 の `BitVec` 仕様間の等価性が形式的に証明済み

ユーザーには2つのパスがあります：効率的な計算にはRadix型を使用、またはMathlibによる推論のために `BitVec` に落とす。

### 5. 階層化APIデザイン

境界チェックや妥当性チェックを含む全モジュールが2段階のAPIを提供：

| 段階 | スタイル | コスト | 使用場面 |
|------|-------|------|----------|
| **Tier 1** | 証明付き：呼び出し元が前提条件の証明を提供 | ランタイムコストゼロ（証明は消去） | コンパイル時に境界が既知 |
| **Tier 2** | チェック付き：`Option` を返すか、ラップ/飽和 | 最小ランタイムコスト（分岐1回） | ランタイムで境界が決定 |

例：
```lean
-- Tier 1: ゼロコスト、証明が必要
def Buffer.readU8 (buf : Buffer) (offset : Nat) (h : offset < buf.size) : UInt8

-- Tier 2: 便利、Option を返す
def Buffer.checkedReadU8 (buf : Buffer) (offset : Nat) : Option UInt8
```

### 6. 名前付き信頼仮定（C-004a）

外部世界についての全公理は厳格な規律に従います：

- `axiom` 宣言でなければならない（`opaque` や `constant` ではない）
- `trust_` プレフィックスを使用
- `Prop` 型でなければならない
- 仮定の根拠となる外部仕様（POSIX、ARMマニュアル等）を引用
- 専用の `Assumptions.lean` ファイルに収集
- 監査可能：各 `trust_*` 公理は外部動作についての明確でレビュー可能な主張

### 7. Lean 4 デフォルト以外の数学的公理なし（C-004）

Radixはカスタム数学的公理を導入しません。Layer 1 の `trust_*` 公理は物理世界（OS動作、ハードウェアセマンティクス）についての主張であり、数学的主張ではありません。この区別は根本的です：数学的公理は証明の健全性にグローバルに影響する。外部世界の公理は局所的な信頼仮定です。

### 8. モジュラー独立性

モジュールは独立して使用できるよう設計されています：

- `Word` は内部依存関係ゼロ — 基盤
- 各モジュールは個別にインポート可能（`import Radix.Word`）
- 依存関係は厳密に上方向に流れる：`Word → Bit → Bytes → Memory → Binary`
- `Concurrency` と `BareMetal` は相互依存のない独立モデルモジュール

### 9. シフトカウント正規化（FR-002.1a）

全シフト・回転操作は `count % bitWidth` で正規化し、Rustのセマンティクスに合致：

- Cは範囲外シフトの動作を**未定義**とする
- Rustは正規化する（カウントをラップ）
- Radixは Rust に従う：全入力に対して定義済みの予測可能な動作

### 10. 完全な符号付き除算処理（FR-001.2a/b）

`MIN / -1` のエッジケース（符号付きオーバーフロー）は4つの算術モード全てで明示的に処理：

| モード | `MIN / -1` の結果 | `MIN % -1` の結果 |
|------|-------------------|-------------------|
| ラッピング | `MIN`（ラップ） | `0` |
| 飽和 | `MAX`（クランプ） | `0` |
| チェック付き | `none` | `none` |
| オーバーフロー付き | `(MIN, true)` | `(0, true)` |

これは `Word.Spec` で形式的に仕様化され、`Word.Lemmas.Overflow` で全符号付き型について証明済み。
