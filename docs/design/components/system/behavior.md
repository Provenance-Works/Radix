# System Module: Behavior Specification

Version: 0.1.0-draft
Last updated: 2026-03-07

## 1. 組み込みIO APIの利用 (Lean Native IO Operations)

### 1.1 FFI依存の完全排除
Lean 4コンパイラでのシステムの操作において、C言語の関数(extern)を経由することは完全に禁止されています(ADR C-001)。
そのため、sysOpenなどの関数は、Lean 4の標準ライブラリ（IO.FS 等）にその役割を完全に委譲します。C言語の層を持たないため、GCとOSリソースのライフサイクル間のインピーダンスミスマッチ、UAF、メモリリークのリスクを完全に排除できます。

### 1.2 IO モナドによる順序保証 (Monadic I/O Sequence)
OSレベルのデータ変更アクセスを行うすべての機能は、副作用を適切にカプセル化するために必ず IO α 型を返します。Lean の副作用は一貫して RealWorld トークンを通してシリアライズされます。

## 2. エラー処理の振る舞い (Error Recovery Behavior)

### 2.1 例外フリー (Exception-less execution)
Except SysError α がすべての標準的システムコール失敗時の返り値として使用されます。Lean4標準の例外スローキャッチ機構 (throw, try-catch) を内部で捕捉し、APIレベルでは直和型 (Except) にすることで、関数が失敗したか成功したかを型シグネチャ上で完全に制御し、型検査時に全てのフェイルステート処理を強制します。生のプラットフォーム依存のエラーコード（マジックナンバー等）は露出させません。

### 2.2 メモリリソースリークの防止対策
リソース枯渇（Resource Exhaustion）問題への対策として、GCファイナライザの遅延による自動クローズに過度に依存することは危険です（FDリサイクル攻撃枯渇リスク）。そのため、ファイルディスクリプタ等の有限のOSリソースは、原則的にモナディックなスコープ付きライフタイム管理（ブラケットパターン: withFile）で取得と即時解放を同期させる設計を強制します。

## 3. 定理と証明におけるシステムレイヤーの扱い (Proof Handling in System Layer)

OS層は「不確実性（Unpredictable state）」を持つため、System モジュールに対する完全な事前証明を構築することは原理的に不可能です。
すべてのOS機能やIOは理論的な予測が不可能であることを前提とし、実行時フォールバック機構（エラーハンドリング）を通じてのみ作用を取り出します。
